# マッチングロジックの再構築

## 1. 調査の理由（動機）

現在、私は日本でCOBOLおよびVB.netを用いた基幹システムの保守・運用を担当しています。
長年運用されているレガシーシステムにおいて、最も頻繁に登場し、かつパフォーマンスに影響を与えるのが「2つのデータのマッチング処理」です。

現場のシステムでは２つのデータを同じ基準でSORTした後、１行ずつ突き合わせることで確認を行っていますが、メモリー面ではとてもいいと思いますが、処理時間が長く非効率的と思いました。
そのため、最新のJava機能を活用し、いかにして「可読性・保守性・パフォーマンス」を同時に向上させるかを考察するため、本調査を行いました。

## 2. Java 17へのマイグレーションにおける検討観点

今回の調査では以下の3点を中心に再設計を行いました。

- **データ構造の刷新**: ファイルI/Oを減らし、メモリ上のHashMapを活用することで探索速度を短縮
- **Java 17の最新機能**: Recordによるデータ不変性の確保と、Stream APIによる宣言的なロジック記述
- **リソースの有効活用**: 現場ではマッチングの為に何回もソートをしていましたが事前にソートする工程を排除

## **3. コード比較（COBOL vs Java 17）**

### **【修正前】**COBOL：順次突き合わせロジック

※ 事前にファイルをキー順にSORTしておく必要があります。

```plain
* 2つのファイルを読み込みながらループ
PERFORM UNTIL END-OF-FILE
    EVALUATE TRUE
        WHEN IN1-KEY = IN2-KEY
            PERFORM MATCHED-PROCESS   *> マッチング成功
            PERFORM READ-IN1, READ-IN2
        WHEN IN1-KEY < IN2-KEY
            PERFORM ONLY-IN-DB-PROCESS *> DBにのみ存在
            PERFORM READ-IN1
        WHEN IN1-KEY > IN2-KEY
            PERFORM NEW-DATA-PROCESS   *> 新規受信データ
            PERFORM READ-IN2
    END-EVALUATE
END-PERFORM.
```

### 【修正後】Java 17：Mapベースのマッチング

```plain
// データ構造をRecordで定義
public record UserData(String id, String name, String info) {}

public void matching(List<UserData> dbList, List<UserData> incomingList) {
    // DBデータをMapに格納 (O(n))
    Map<String, UserData> dbMap = dbList.stream()
            .collect(Collectors.toMap(UserData::id, Function.identity()));

    // 新規データの処理
    incomingList.forEach(incoming -> {
        UserData existing = dbMap.remove(incoming.id()); // 取得と同時にMapから削除
        if (existing != null) {
            processMatched(incoming); // [MATCHED]
        } else {
            processNew(incoming);     // [NEW DATA]
        }
    });

    // Mapに残ったものはDBにのみ存在するデータ
    dbMap.values().forEach(this::processOnlyInDb); // [ONLY IN DB]
}
```

## 5. 中間考察：パフォーマンスとリソースのトレードオフ

Java 17へのマイグレーションにおいて、ポイントは「処理時間の短縮」と「メモリ消費量の増加」のトレードオフでした。

今回の刷新により、以下の成果が得られました。

- **処理時間の短縮**: 修正前と比べて９割に処理時間が短縮できた。
- **運用コストの削減**: 従来必須だった「外部SORT工程」が不要になり、ジョブフローが単純化されました。

しかし、すべてのデータを一度にメモリへ展開すると、膨大なギガバイト級のデータを扱う際に OutOfMemoryError のリスクが生じます。そこで、メモリ消費を抑えながらも大容量データにも対応できるよう「分割処理」を加えてを取り入れた実験コードを作成しました。

## 6. 大容量データ対応：分割処理の実装

メモリ使用量を一定に保つため、FILE_2（受信データ）を一定件数ずつ読み込み、DBデータと照合するロジックです。

```plain
// Java 17: Recordを活用したデータ定義
public record UserData(String id, String name, String info) {}

public void processInChunks(List<UserData> incomingData) {
    int chunkSize = 10000;

    // 全データを一度に処理せず、サブリスト（Chunk）に分けてループ
    for (int i = 0; i < incomingData.size(); i += chunkSize) {
        int end = Math.min(i + chunkSize, incomingData.size());
        List<UserData> chunk = incomingData.subList(i, end);

        // このChunkに対してのみマッチングを実行
        executeMatching(chunk);

        // 各Chunk処理後に明示的にメモリ管理の最適化を考慮（GCの負荷軽減）
        System.out.println("Processed up to: " + end + " records...");
    }
}
```

## 8.最終結論

Java 17へのマイグレーションすることで最終的に以下の成果が得られられると確認できました。

- **処理時間の短縮**：修正前と比べて９割近く処理時間を飛躍的な短縮ができました。
- **開発生産性の向上**：COBOLのコードと比較して短縮されて可視性と管理性が上がった。
- **運用コストの削減**: 従来必須だった「外部SORT工程」が不要になり、ジョブフローが単純化されました。
