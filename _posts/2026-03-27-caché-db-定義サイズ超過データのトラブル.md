自分が現在勤めている現場ではCaché DB（InterSystems IRIS）を利用しています。今回は業務の中で身で感じたCaché DBのトラブルについて記述しようと思います。

## Caché DB

Caché DBは他のRDBMS（Oracleなど）とは異なるユニークな特徴があります。

- **多次元データ構造：**内部的には「グローバル」と呼ばれる多次元配列でデータを保持しており、非常に高速で柔軟です。
- **スキーマの柔軟性：**SQLインターフェースを提供していますが、根本はオブジェクト指向データベースに近い性質を持っています。そのため、「型定義に対して比較的寛容」という側面があります。
- **データの格納：**定義された長さを超えるデータが入力された際、RDBMSのように厳格にエラーを出す設定も可能ですが、デフォルトや設定次第では「定義を超えた長さのまま」内部に保存できてしまうことがあります。

## 発生した問題 (事象)

- **環境:** 項目定義が「20バイト（全角10桁相当）」のテーブル。
- **操作:**

    1.  当該項目に「全角11桁（22バイト）」のデータを `INSERT`。
    2. **DBツール（CSE, A5:SQL等）での確認:** 問題なくデータ登録が行われ、`SELECT` 結果も表示される（ただし、表示上は20バイト分＝10桁でカットされて見える）。
    3. **アプリケーション（VB.net等）での実行:** プログラムからDBに接続し、該当データを読み取ろうとした瞬間に、バッファオーバーフローやデータ不整合によるエラーが発生し、正常に処理が継続できない。

## 発生原因

この問題の根本的な原因は、**「入力データの確認漏れ」、「DBエンジン側の許容設定」、「接続ドライバ（ODBC）の厳格なバッファ管理」の差**と認識しています。

- **入力データの確認漏れ：**事前に顧客との入力データの打合せで想定以上のデータの確認チェックが漏れたこと
- **DB側の挙動：**Caché DBのSQLエンジンは、デフォルト設定によっては `MAXLEN` を超えるデータの `INSERT` を厳密にブロック出来てなかったこと。
- **管理ツールの挙動：**CSEやA5:SQLなどのツールは、グリッド表示の際にデータを適切にトリミングして表示するため、裏側に「定義外のデータ」が隠れていることに気づけなかった。
- **プログラム側の挙動：**VB.netなどのプログラムがDBドライバを通じてデータを取得する際、ドライバはDBのメタデータに基づいてメモリ上のバッファを確保します。しかし、実際にDBから送られてくるデータが22バイトあるため、**確保したメモリ領域をデータが溢れ出し（Buffer Overflow）**、ランタイムエラーが発生した。

## 根本対策：不正データの入力を未然に防ぐ

そもそも定義外のデータが入らないように制約を強化する必要があります。クラス定義（Property）の制約強化

### ① クラス定義（Property）の制約強化

```plain
// MAXLENを超えた場合に保存を失敗させる（デフォルトの検証ロジックを活かす）
Property TargetColumn As %String(MAXLEN = 20);
```

### ② アプリケーション層でのバリデーション

```plain
' Shift-JIS(全角2バイト)基準でチェックする場合
Dim sjis As System.Text.Encoding = System.Text.Encoding.GetEncoding("Shift-JIS")
If sjis.GetByteCount(inputData) > 20 Then
	Throw New Exception("入力データが20バイトを超えています。")
End If
```

## 異常データの検知と修正

### 検知：隠れた問題レコードを探す

```plain
-- 文字数ベースで全角10文字（20バイト相当）を超えているものを抽出
SELECT ID, TargetColumn, CHARACTER_LENGTH(TargetColumn) 
FROM Table名 
WHERE CHARACTER_LENGTH(TargetColumn) > 10;
```

### 修正：不正データの正常化

```plain
-- 20バイト（全角10文字）を超えるデータを、先頭10文字に切り詰めて更新
UPDATE Table名
SET TargetColumn = SUBSTRING(TargetColumn, 1, 10) 
WHERE CHARACTER_LENGTH(TargetColumn) > 10;
```

## まとめ

Caché DBの「柔軟性」は大きなメリットですが、古い言語から移行した環境や、厳密な型定義を持つVB.net等と組み合わせる際は、この特性が仇となることがあると思いました。**「DBで正常に見えている」状況でも、プログラムに不具合を発生するデータの状態**になっている可能性があるということを経験しました。
